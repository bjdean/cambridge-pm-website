On Fri, Apr 12, 2002 at 03:29:10PM +0100, Tim Cutts wrote:
> On Fri, Apr 12, 2002 at 10:49:19AM +0100, Peter Clay wrote:
> > What's the most efficient way of determining whether a scalar contains an
> > integer or a string?
> This works (and there's an equivalent for floating point numbers):
> use POSIX;
> $!=0;
> ($num, $unparsed) = &POSIX::strtol($str, 10);
> if ($!) { print "Not an integer\n" };
> This has a few advantages over regexps, such as the ability to specify
> whether the scalar is a number in a particular base, while still
> maintaining locale sensitivity.
> 
> I don't find this particularly useful for integers, but the floating
> point equivalent is a lot less cumbersome than the regexp required for
> floating point...

I don't know what Pete actually wanted, whether it was "what perl was
actually storing", or whether it was "is this a valid <foo>"

In fact, I was wrong, as I'm sure NWC spotted, you actually need an
SvIOKp(SV*) and SvPOKp(SV*) if you want perl to just tell you the
value of the flag, and not promote the SV* to the relevant subtype
based SV structure. This is in man perlguts around the
"What's Really Stored in an SV?" subheading for anyone who *actually*
cares.

Other than that, however, I think I find the above way neatest to determine
whether an SV* is an integer. Obviously anything in Perl can be cast to a
string. [1]

[1] By saying this, someone will jump out and show me a counter-example,
    but I can't think of one as I type this.
