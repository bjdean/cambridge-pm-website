>Soooo. I've tried putting it all in a hash, but there's too much data.
>My current mediocre approach is to go throught the file once for each gene
>and print to n intermediary files (where n is total number of genes), which
>frankly takes far too long.  Shall swap beer for useful ideas.

Use Sleepycat libdb? 
http://search.cpan.org/search?module=BerkeleyDB

This should be able to handle hashes bigger than RAM size. If not,
it's broken. The libdb C API certainly allows it.

Alternatively, sort your file by gene1, and keep a hash of offsets into
the file by gene1. When looking for a (gene1,gene2,score) tuple, jump
to the appropriate place in the file for gene1, and do a sequential scan
from that point. The OS buffer cache should help.
