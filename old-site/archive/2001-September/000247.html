On Mon, Sep 24, 2001 at 11:06:46AM +0100, Matthew Garrett wrote:
> 
> On topic, though - I've never used Perl for any terribly serious
> scientific apps, so have absolutely no idea how useful it is for
> performing reasonably computationally heavy tasks. The program in question
> is effectively picking random numbers, doing lookups in a table based on 
> those and then picking another random number to see whether what it's got 
> out of the table actually happens or not. If it does, it does weird shit 
> (tm) to an array and then goes back to step 1. The configuration file 
> parsing currently seems to be a nightmare and would probably be reason 
> enough to reimplement things from scratch, but it already takes upwards of 
> 12 hours for some runs and I don't really want to find that becoming 
> significantly larger. Is this sort of thing feasible?

One possibility is of course to leave the nasty CPU intensive stuff in
some other language and then call it from a scriptable language like
perl.  Whether you do this as something like a swig of XS perl module,
or embed some other interpreter such as tcl in your C/C++ program is
more or less up to you.

Although I haven't written any code this way ye, I use a number of
scientific apps that do work this way, and they're a joy to automate as
a result - e.g tcl scripts driving automated DNA sequence assembly with
the Staden package.

It's quite a bit more painful to do this sort of thing with
non-scriptable alternatives like phrap, which is a shame because it's a
better assembly engine.
